# Product Dev Full Cycle

按以下 6 个阶段执行，并在每阶段产出可交付物。始终先确认范围、再执行、再验收。

## 执行总则

- 先收敛目标：定义用户、场景、成功指标与截止时间。
- 小步交付：每阶段都给出可审阅产物，不跳阶段。
- 门禁推进：未通过当前阶段验收标准，不进入下一阶段。
- 迭代优先：优先交付可运行最小版本，再做增强。

## 阶段 1：需求梳理

输出：`PRD`、功能清单、优先级（P0/P1/P2）

执行：
- 明确产品目标、用户痛点、核心场景。
- 收集需求并去重，定义范围内/范围外。
- 用影响力 × 成本排序，确定首版范围。

验收：
- `PRD` 包含目标、用户故事、验收标准、里程碑。
- 功能列表每项都有优先级与业务价值。

## 阶段 2：产品设计

输出：产品原型、UI 设计稿、交互流程图

执行：
- 绘制关键页面和核心任务路径。
- 定义页面状态：空态、加载、错误、成功。
- 明确交互规则与边界行为。

验收：
- 核心链路可从入口走到完成态。
- 每个页面有清晰输入、输出与跳转条件。

## 阶段 3：技术开发

输出：可运行代码、接口文档、数据库脚本

执行：
- 搭建开发环境，定义前后端边界与契约。
- 先实现主流程，再补充异常分支。
- 同步编写 API 说明、环境变量与迁移脚本。

验收：
- 本地可启动并演示主流程。
- API 可复现调用，数据模型与脚本一致。

## 阶段 4：测试验收

输出：测试报告、Bug 清单、修复后的可交付版本

执行：
- 执行功能测试、兼容性测试、性能基线测试。
- 记录缺陷：复现步骤、影响范围、修复优先级。
- 修复后做回归验证。

验收：
- P0/P1 缺陷关闭。
- 关键路径通过率达到约定阈值。

## 阶段 5：上线部署

输出：正式上线版本、可访问域名、服务器配置文档

执行：
- 准备发布清单与回滚策略。
- 配置部署环境、域名与监控告警。
- 执行发布并做上线后冒烟。

验收：
- 线上可用，核心指标正常，回滚方案可执行。

## 阶段 6：运营迭代

输出：运营数据报告、迭代需求清单、版本更新计划

执行：
- 监控关键指标并收集用户反馈。
- 基于数据更新优先级，规划下一迭代。
- 维护变更记录与版本说明。

验收：
- 下一周期需求池已排序。
- 形成“数据 -> 问题 -> 方案 -> 发布”的闭环。
